# 実装フロー

> 機能実装の進め方を定義する。  
> **エピック単位で計画し、タスク単位で実行する**。

---

## 全体像

- **Epic**（例: 1. 認証・アカウント管理）
  - **Phase 0. デザインフェーズ** — エピック全体のUIをモック（.pen）→ デザイントークン確認
  - **US-1** ストーリー
    - `[TASK]` タスクA　← 実行単位
    - `[TASK]` タスクB
  - **US-2** ストーリー
    - `[TASK]` タスクC
  - **Phase 3. 完了確認** — Epic 全体の受け入れ基準を確認 → main へマージ

---

## Phase 0. エピック開始前：デザインフェーズ

### 目的

エピック内の全ページ・全UIパターンを **実装前に俯瞰し、デザイントークンと要素を揃える**。  
モックで詰めることで、タスク単位の実装が迷いなく進む。

### やること

#### 1. エピック内のページ・画面を洗い出す

`docs/issues/<epic>/epic.md` とストーリー一覧からUIが必要な画面を列挙する。

```
例: 1. 認証・アカウント管理
  - ログインページ
  - OAuthコールバック（エラー画面）
  - プロフィール表示
```

#### 2. .pen でモックを作る

**モックで確認・決定すること：**

| 項目                         | 内容                                           |
| ---------------------------- | ---------------------------------------------- |
| ページレイアウト             | ヘッダー・サイドバー・コンテンツ領域の配置     |
| 情報構造                     | 何をどの順で見せるか                           |
| UIパターン                   | カード・リスト・フォーム等どのパターンを使うか |
| 新規コンポーネント           | Park UI にないカスタムコンポーネントの洗い出し |
| 空・ローディング・エラー状態 | 画面状態のパターン                             |

**モックの粒度：** ピクセルパーフェクトは不要。「配置と要素が伝わるワイヤーフレーム」で十分。

#### 3. デザイントークンを確認・調整する

モックを見ながら `panda.config.ts` の設定を確認し、不足があれば追加する。

```ts
// panda.config.ts に追加が必要なら
theme: {
  extend: {
    semanticTokens: {
      colors: {
        // エピックで必要になった意味的トークンをここで定義
      },
    },
  },
}
```

トークンをここで固めることで、エピック内のコンポーネントが同じ値を参照できる。

> ⚠️ この段階でトークンを「だいたい決める」のが目的。  
> タスク実装中に細かい調整は起きるが、カラーパレット・タイポグラフィの基準はここで合意する。

#### 4. コミット

```bash
git commit -m "design(<epic>): モックデザイン・デザイントークン確認"
```

## Phase 1. ストーリー → タスクの確認

`docs/issues/<epic>/<story>/story.md` を読み、タスク一覧と依存関係を整理する。

```bash
ls "docs/issues/<epic>/<story>/*.md"
```

依存関係のないタスクは**並列実行**できる（サブエージェントを活用）。

---

## Phase 2. タスク単位の実装（実行単位）

**タスク1件 = ブランチ1本 = PR1本**。

### ブランチ作成

```bash
# epic・story ブランチが既に存在する前提で
git checkout us/<story-name>
git checkout -b task/<task-name>
```

### フロントエンドタスク

`frontend-workflow` スキルに従う。

```
E2Eテスト作成（Red）
  → 実装（Green）
  → browser-use で視覚確認
  → Refactor
```

> Phase 0 のモックを見ながら実装する。迷ったらモックに立ち返る。

### バックエンドタスク

`backend-workflow` スキルに従う。

```
TDD: Red → Green → Refactor
```

### タスク完了チェックリスト

- [ ] `[TASK]*.md` のチェックリストを更新した
- [ ] テストが通っている
- [ ] (フロント) browser-use で視覚確認した
- [ ] コミットが小さく意味のある単位になっている

```bash
git commit -m "<type>(<scope>): <変更内容>"
```

## Phase 3. エピック完了確認

`epic.md` の受け入れ基準（`## 受け入れ基準`）をすべて確認する。

- [ ] Must ストーリーがすべて Done
- [ ] E2E シナリオが通過
- [ ] ドキュメント更新済み

## デザイン統一のために

JIT（その場その場でデザインを決める）だとエピックをまたいでUIがバラバラになるリスクがある。  
これを防ぐための構造：

| レイヤー           | 統一の担保                                         |
| ------------------ | -------------------------------------------------- |
| **トークン**       | `panda.config.ts` で全コンポーネントが同じ値を参照 |
| **エピック**       | Phase 0 のモックで「このエピックのUIルール」を確認 |
| **コンポーネント** | `app/components/ui/`（Park UI）と `common/` を共有 |

**エピックをまたぐ場合：** 複数エピックで同じUIパターンが出てきたら `common/` に抽出する。  
抽出のタイミングは「2回目に使おうとしたとき」（= Rule of Three の簡易版）。
