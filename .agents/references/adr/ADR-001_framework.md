# ADR-001: フルスタックフレームワークに Nuxt 4 を採用する

## Context

Java製の記事投稿サイト（knowledge）をモダンな技術スタックにリライトするにあたり、フロントエンド・バックエンドのフレームワークを選定する必要があった。

主な要件：

- フロントとバック間で**型を共有**したい（モノレポ）
- **Vue.js** を使いたい
- 記事ページの **SEO対応**（SSR）が必要
- 開発・保守コストを下げたい（自作フレームワークからの脱却）

## Candidates

| 候補 | 概要 |
|------|------|
| Nuxt 4 フルスタック | Vue 3 + Nitro サーバーが一体。`shared/` で型が自動共有される |
| Hono + Vue 3 モノレポ | Hono で API、Vue 3 で UI。`packages/types/` で型共有 |
| Nuxt 4 + 別バックエンド | Nuxt をフロントのみに使い、APIは別サービスで立てる |

## Decision

**Nuxt 4 フルスタック**を採用する。

## Rationale

- `shared/` ディレクトリがサーバー・クライアント双方に自動公開されるため、Zod スキーマを一度定義するだけで型共有が完結する
- SSR がビルトインで、記事ページの SEO 対応に追加設定が不要
- API サーバーを別途立てる必要がなく、単一リポジトリで完結する
- Nitro（H3）は軽量かつ本番実績があり、将来的に Edge デプロイも可能

## Consequences

- dev サーバーの起動がやや重い（Nuxt 3 より改善されているが、Hono 単体より遅い）
- バックエンドのロジックが極めて複雑になった場合、Nitro の薄さが制約になる可能性がある（その場合は API サーバーを分離する）
- Nuxt 4 は Nuxt 3 からの破壊的変更があるため、`app/` ディレクトリ構造に慣れる必要がある
