## コーディング

- TDDで必ずテストコードを先に書いてから実装すること(Red-Green-Refactorサイクル)
- 作業はfeatureブランチをmainから切って行うこと
- コミットは小さく、意味のある単位で行うこと
- コミットが完了したらCHANGELOG.mdを更新すること
- デザインは.pen形式で行うこと

  ```md
  ## yyyy-mm-dd

  ### [type]

  - <変更内容>
  ```

## 開発の進め方

### 前提

- github issueとローカルの`docs/issues`の内容を連携させている
- 連携には`.claude/hooks/update-issues.sh`を使用している。
- それぞれのステップが終わるためにコミットを行い、ユーザーにpull requestのレビューを依頼すること。

### フロントエンドの開発フロー

1. **モックデザインの作成** — `.pen` 形式で画面レイアウト・コンポーネント構成を設計する
2. **E2E テストの作成（Playwright）** — モックをもとに期待する操作・表示をテストとして記述する（Red）
3. **実装** — テストが通るように実装し、`browser-use` で実際の表示を確認する（Green → Refactor）

### バックエンドの開発フロー

1. **テストコードの作成** — ユースケース・ドメインロジックをテストで先に定義する（Red）
2. **実装** — テストが通るように実装する（Green）
3. **DDD 観点レビュー** — ドメインモデル・集約・境界の妥当性を確認し、必要なら設計を修正する（Refactor）

## テスト実装方針

- テストはユースケース単位で整理する（正常系・異常系での分類はしない）
- エッジケースは書く。ただし実際に起こりうるものに限定し、細かすぎるケースは書かない
- テストは AAA パターン（Arrange / Act / Assert）で書く
- テストタイトルは Given-When-Then パターンで書く

```ts
// Given: <前提条件>, When: <操作>, Then: <期待結果>
it("Given 未ログインユーザーが, When 記事を投稿しようとすると, Then 認証エラーが返る", () => {
  // Arrange
  // Act
  // Assert
});
```

## コメント方針

| 状況                                             | 方針                                     |
| ------------------------------------------------ | ---------------------------------------- |
| 読めばわかるコード                               | コメントなし（変数名・関数名で表現する） |
| 「なぜ」この実装にしたか（制約・バグ回避・意図） | インラインコメントで補足する             |
| composable・ユーティリティ関数の公開 API         | JSDoc (`/** */`) を書く                  |
