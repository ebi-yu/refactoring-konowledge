# STORY テンプレート

> ユーザーが価値を感じる 1 機能単位で起票する Issue のテンプレート。
> `01_feature_scope.md` のテーブル 1 行（機能ID）が 1 STORY に対応する。

---

## タイトル形式

```
(US-XX) <機能ID> <機能名>
例: [US-1] メールアドレス登録・ログイン
```

## ラベル

`type:story`

---

## Issue 本文

```markdown
## ユーザーストーリー

<!-- ⚠️ Who / What / Why はすべて必須。空欄のまま起票しないこと。 -->

| 項目                   | 内容                                                                 |
| ---------------------- | -------------------------------------------------------------------- |
| **Who**（誰が）        | 〈ユーザーロール〉例: 登録済みユーザー、管理者                       |
| **What**（何をしたい） | 〈目的・やりたいこと〉例: メールアドレスとパスワードでログインしたい |
| **Why**（なぜ・価値）  | 〈得られる価値〉例: 自分の記事・設定に安全にアクセスしたいから       |

**ストーリー文（上記を文章化）:**

> **〈Who〉** として、**〈What〉** がしたい。なぜなら **〈Why〉** だからだ。

## 背景・補足

<!-- スコープ表の「備考」列の内容や、UX 上の注意点を記載 -->

## 受け入れ基準

> Given / When / Then 形式で記述する。

- [ ] **Given** <前提条件>  
       **When** <操作>  
       **Then** <結果>

- [ ] **Given** ...  
       **When** ...  
       **Then** ...

## スコープ・制約

| 項目               | 内容                        |
| ------------------ | --------------------------- |
| **優先度**         | `Must` / `Should` / `Could` |
| **旧システム対応** | ✅ 存在 / 🆕 新規           |
| **スコープ**       | `[x]` 実装 / `[△]` 部分実装 |
| **部分実装の制限** | （`[△]` の場合のみ記載）    |

> **Must**: このリリースに必須 / **Should**: できれば含める / **Could**: 余裕があれば

## タスク一覧

> STORY を技術タスクに分解し、別 Issue（TASK）として起票してリンクする。

- [ ] #<TASK Issue 番号> - <TASK タイトル>
- [ ] #<TASK Issue 番号> - <TASK タイトル>

## 参照

- EPIC: #<EPIC Issue 番号>
- [[01_feature_scope]] - 機能ID: X-X
```
