# バックエンド開発フロー

このプロジェクトのバックエンドは **Nuxt 4 server routes + Prisma + DDD（ドメイン駆動設計）** で構成される。
実装は必ず以下の3ステップで進める。

## ブランチ戦略

`.claude/references/branch-strategy.md` を参照。

---

## 実装ステップ

各ステップが完了したら**コミットを作成してユーザーにマージを依頼**し、マージされてから次のステップに進む。

### Step 1. テストコードの作成（Red）

ユースケース・ドメインロジックをテストで先に定義する。
「何を達成すべきか」をテストで明確にしてから実装に入る。

```ts
// Given: <前提条件>, When: <操作>, Then: <期待結果>
it("Given 未ログインユーザーが, When 記事を投稿しようとすると, Then 認証エラーが返る", () => {
  // Arrange
  // Act
  // Assert
});
```

テスト整理の方針:

- ユースケース単位で整理する（正常系・異常系での分類はしない）
- エッジケースは書く。ただし実際に起こりうるものに限定し、細かすぎるケースは書かない

**✅ Step 1 完了時:**

```bash
git add .
git commit -m "test: <ストーリー名> のテストコード作成（Red）"
# PR を作成して us/<story-name> へのマージをユーザーに依頼
# → マージ確認後に Step 2 へ進む
```

---

### Step 2. 実装（Green）

テストが通るように最小限のコードを書く。
過剰な実装・先読み設計は避け、まずテストを通すことに集中する。

コメント方針:

| 状況                                             | 方針                                     |
| ------------------------------------------------ | ---------------------------------------- |
| 読めばわかるコード                               | コメントなし（変数名・関数名で表現する） |
| 「なぜ」この実装にしたか（制約・バグ回避・意図） | インラインコメントで補足する             |
| composable・ユーティリティ関数の公開 API         | JSDoc (`/** */`) を書く                  |

**✅ Step 2 完了時:**

```bash
git add .
git commit -m "feat: <タスク名> の実装"
# PR を作成して us/<story-name> へのマージをユーザーに依頼
# → マージ確認後に Step 3 へ進む
```

---

### Step 3. DDD 観点レビュー（Refactor）

ドメインモデル・集約・境界コンテキストの妥当性を確認し、必要なら設計を修正する。
「動く」だけでなく「正しくモデリングされているか」を確認するステップ。

確認観点:

- ドメインオブジェクトの責務が適切か
- 集約の境界が正しいか（トランザクション整合性）
- ユースケースとドメインロジックが分離されているか
- Prismaのスキーマ変更が必要な場合は `bunx prisma db push` で反映する

**✅ Step 3 完了時:**

```bash
git add .
git commit -m "refactor: <タスク名> の DDD 設計見直し"
# PR を作成して us/<story-name> へのマージをユーザーに依頼
# → 全タスクが us にマージされたら epic/<epic-name> への PR を作成
```
