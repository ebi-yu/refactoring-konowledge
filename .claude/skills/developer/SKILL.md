# developer スキル

このプロジェクトの開発フローは **エピック単位で計画し、タスク単位で実行する**。

```
Epic
├── Phase 0. デザインフェーズ  ← UIモック・デザイントークン確定
├── US-1（ストーリー）
│   ├── [TASK] タスクA
│   └── [TASK] タスクB
├── US-2（ストーリー）
│   └── [TASK] タスクC
└── Phase 3. 完了確認          ← 受け入れ基準をすべて確認 → main マージ
```

GitHub Issues とローカルの `docs/issues/` を連携し、Epic → Story → Task の3層でタスクを管理する。

---

## 開発の進め方

### 前提

- GitHub Issue とローカルの `docs/issues/` の内容を連携させている
- 連携には `.claude/hooks/update-issues.sh` を使用する
- フロントエンドとバックエンドの実装フローは Step 3 の「タスク種別ごとの実装フロー」を参照

---

### Step 0. ブランチの作成（実装前に必ず行う）

上位ブランチから順に作成する。

```bash
# 1. Epic ブランチ（main から）
git checkout main
git checkout -b epic/<epic-name>

# 2. Story ブランチ（epic から）
git checkout -b us/<story-name>

# 3. Task ブランチ（us から）
git checkout -b task/<task-name>
```

既に Epic・Story ブランチが存在する場合は、そこから切る。

---

### Step 1. エピックの確認

`docs/issues/` 配下のディレクトリを確認し、作業対象のエピックをユーザーに確認する。

```bash
ls docs/issues/
```

Epic・US・Taskのファイルはすでに存在している前提で進む。

---

### Phase 0. エピック開始前：デザインフェーズ

実装に入る前に、エピック内の全UIを俯瞰してデザイントークンと要素を揃える。  
モックで詰めることで、タスク単位の実装が迷いなく進む。

#### 1. エピック内の画面を洗い出す

`docs/issues/<epic>/epic.md` とストーリー一覧からUIが必要な画面を列挙する。

#### 2. .pen でモックを作る

`frontend-workflow` スキルの Pencil MCP を使い、ワイヤーフレームレベルのモックを作成する。

確認・決定する項目：

| 項目                         | 内容                                           |
| ---------------------------- | ---------------------------------------------- |
| ページレイアウト             | ヘッダー・サイドバー・コンテンツ領域の配置     |
| 情報構造                     | 何をどの順で見せるか                           |
| UIパターン                   | カード・リスト・フォーム等どのパターンを使うか |
| 新規コンポーネント           | Park UI にないカスタムコンポーネントの洗い出し |
| 空・ローディング・エラー状態 | 画面状態のパターン                             |

ピクセルパーフェクトは不要。「配置と要素が伝わるワイヤーフレーム」で十分。

#### 3. デザイントークンを確認・調整する

モックを見ながら `panda.config.ts` の設定を確認し、不足があれば追加する。

```ts
theme: {
  extend: {
    semanticTokens: {
      colors: {
        // エピックで必要になった意味的トークンをここで定義
      },
    },
  },
}
```

カラーパレット・タイポグラフィの基準はここで合意する。タスク実装中の細かい調整は許容するが、基準はこの段階で固める。

#### 4. コミット

コミットの前に、モックとトークンの内容をユーザーに見せて確認を取る。OKをもらってからコミットする。

```bash
git commit -m "design(<epic>): モックデザイン・デザイントークン確認"
```

---

### Step 2. ユーザーストーリーとタスクの確認

エピックに紐づくストーリー（`[US-N]` ディレクトリ）を確認し、実装するストーリーを選ぶ。
ストーリー内のタスク（`[TASK]*.md`）を読み、依存関係を整理する。

---

### Step 3. タスクの並列実行

依存関係のないタスクは**サブエージェントで並列実行**する。

```
タスクA ──► サブエージェント1
タスクB ──► サブエージェント2  （同時起動）
タスクC ──► サブエージェント3
```

#### サブエージェントの起動方法

Agent ツールを使い、**1メッセージの中に複数の Agent tool call を並べて**同時起動する。

```
Agent({
  prompt: `
以下のタスクを実装してください。

タスクファイル: docs/issues/<epic>/<story>/[TASK] タスクA.md
スキル: frontend-workflow（または backend-workflow）
ブランチ: task/<task-name>（すでに存在する前提）

タスクファイルの作業内容をすべて完了させ、
完了した項目のチェックボックスを [x] に更新してください。
  `
})

Agent({
  prompt: `
以下のタスクを実装してください。

タスクファイル: docs/issues/<epic>/<story>/[TASK] タスクB.md
...
  `
})
```

各サブエージェントへのプロンプトに必ず含めること:

- 対象の `[TASK]*.md` ファイルパス
- タスクの種別（フロント／バック）
- 作業ブランチ名
- 完了時に `[TASK]*.md` のチェックリストを更新すること

#### タスク種別ごとの実装フロー

**フロントエンドタスク**（Vue/Nuxt/Panda CSS）

```
E2Eテスト作成（Red）→ 実装（Green）→ browser-use で視覚確認 → Refactor
```

- E2E は Playwright で Given-When-Then 形式、ユースケース単位で書く
- 実装後は必ず `browser-use` でレイアウト・トークン・インタラクションを確認する

**バックエンドタスク**（Nuxt server routes / Prisma / DDD）

```
テスト作成（Red）→ 実装（Green）→ DDD観点レビュー（Refactor）
```

- Refactor では「ドメインオブジェクトの責務」「集約の境界」「ユースケースとドメインの分離」を確認する
- Prisma スキーマ変更が必要なら `bunx prisma db push` で反映する

---

### Step 4. タスク完了時の必須手順

各タスクが完了したら**必ず**以下を実行する:

1. `[TASK]*.md` の完了した作業項目を `[x]` に更新する
2. GitHub に同期する:
   ```bash
   bash .claude/hooks/update-issues.sh "<タスクファイルパス>"
   ```
3. コミットを作成する

すべてのタスクが完了したら、`story.md` の作業内容も更新してコミットする。

---

### Step 5. PR の作成とマージ

各ステップ完了後、PR を作成してユーザーにマージを依頼する（マージされてから次のステップへ進む）。

```
task/<name>  →  us/<story-name>   （タスク完了時）
us/<name>    →  epic/<epic-name>  （ストーリー完了時）
epic/<name>  →  main              （Epic完了時）
```

---

### Phase 3. エピック完了確認

すべてのストーリーの PR がマージされたら、`epic.md` の受け入れ基準を確認する。

`docs/issues/<epic>/epic.md` の `## 受け入れ基準` セクションを開き、以下をすべてチェックする：

- [ ] Must ストーリーがすべて Done
- [ ] E2E シナリオが通過
- [ ] ドキュメント更新済み

確認できたら `epic/<name> → main` の PR を作成してユーザーにマージを依頼する。

---

## デザイン統一のために

JIT（その場その場でデザインを決める）だとエピックをまたいでUIがバラバラになるリスクがある。  
Phase 0 でモックとトークンを固めることで、この問題を事前に防ぐ。

| レイヤー           | 統一の担保                                         |
| ------------------ | -------------------------------------------------- |
| **トークン**       | `panda.config.ts` で全コンポーネントが同じ値を参照 |
| **エピック**       | Phase 0 のモックで「このエピックのUIルール」を確認 |
| **コンポーネント** | `app/components/ui/`（Park UI）と `common/` を共有 |

**エピックをまたぐ共通コンポーネント：** 2回目に使おうとしたときに `common/` へ抽出する（Rule of Three の簡易版）。
