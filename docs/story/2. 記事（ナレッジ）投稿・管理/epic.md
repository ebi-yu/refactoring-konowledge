# [Epic] 記事（ナレッジ）投稿・管理

## 概要

ユーザーが Markdown で記事を作成・編集・削除し、公開範囲を制御しながら知識を発信できるコア機能群。
ファイル添付・下書き保存・公開範囲設定を含み、システムの中核となる投稿ワークフローを提供する。
旧システムの記事機能を引き継ぎつつ、Ph1 スコープに絞り込んだ最小完全な実装を目指す。

## Epic ストーリー

**投稿者** として、**Markdown で記事を書いて公開範囲を指定しながら投稿・管理** がしたい。なぜなら **自分の経験やノウハウをチームメンバーにいつでも参照できる形で残したい** からだ。

## スコープ（Story 一覧）

> 優先度: `Must` = リリース必須 / `Should` = できれば含める / `Could` = 余裕があれば

| 優先度 | # | ストーリー（一文） | Issue |
|--------|---|-------------------|-------|
| `Must` | 1 | 投稿者として、タイトルと Markdown 本文を入力して新しい記事を作成したい。なぜなら自分の知識やノウハウをチームメンバーに届けたいから。 | - |
| `Must` | 2 | 投稿者として、mermaid形式で図を書きたい。なぜなら、md記法の範囲内で図を書き、記事を視覚的に分かりやすくしたいから。 | - |
| `Must` | 3 | 投稿者として、公開済み記事の内容を後から編集・更新したい。なぜなら状況が変わったときに正しい情報を読者に届け続けたいから。 | - |
| `Must` | 4 | 投稿者として、不要になった自分の記事を削除したい。なぜなら古くなった情報でチームメンバーが混乱するのを避けたいから。 | - |
| `Must` | 5 | 投稿者として、記事を「公開 / ログインユーザーのみ / 非公開」の3段階で公開範囲を設定したい。なぜなら社外秘の内容や書きかけの記事を誤って外部に見せたくないから。 | - |
| `Must` | 6 | 投稿者として、完成前の記事を下書きとして保存し、後から続きを書いて公開したい。なぜなら時間をかけて丁寧に仕上げた記事をチームに届けたいから。 | - |
| `Should` | 7 | 投稿者として、記事に画像・PDF などのファイルを添付したい。なぜなら図や資料があった方が読んでいる人に内容が伝わりやすいから。 | - |

## スコープ外

- **テンプレート機能**（2-6）: Ph1スコープ外。旧システムにあったが利用率が低く、最小実装から除外
- **共同編集者設定**（2-7）: Ph1スコープ外。複雑な権限管理が必要なため将来のEpicで対応
- **改訂履歴・差分表示**（2-8）: Ph1スコープ外。実装コストに対してPh1では優先度が低い
- **ピン留め**（2-9）: Ph1スコープ外。管理者向け機能として別Epicで検討
- **メールから記事作成**（2-10）: スコープアウト。専用クライアントとの競合・セキュリティリスクがある
- **グループ限定公開**（6-2）: Ph2（Keycloak連携が前提）。この Epicでは扱わない

## 受け入れ基準（Epic 完了の定義）

- [ ] 配下のすべての `Must` Story が Done になっている
- [ ] 記事作成・編集・削除・公開範囲変更・下書き保存・ファイル添付の E2E シナリオが通過している
- [ ] ドキュメント（ADR / API仕様）が更新されている
