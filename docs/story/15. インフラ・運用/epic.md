# [Epic] インフラ・運用（セルフホスト）

## 概要

本システムはセルフホストを前提として設計する。
Docker Compose 一発で起動できる構成を必須とし、環境変数による設定・ヘルスチェック・バックアップ・アップグレード手順を標準提供する。
クラウドストレージ（S3/R2/MinIO）への切り替えも本 Epic で扱う。

## Epic ストーリー

**システム管理者** として、**Docker Compose 一発でシステムを起動して本番環境で安定稼働させたい**。なぜなら **セットアップに何日もかかったり、設定方法が複雑だと運用を任せられる人が限られてしまうから**。

## スコープ（Story 一覧）

> 優先度: `Must` = リリース必須 / `Should` = できれば含める / `Could` = 余裕があれば

| 優先度 | # | ストーリー（一文） | Issue |
|--------|---|-------------------|-------|
| `Must` | 1 | 管理者として、ファイルの保存先を S3 / R2 / MinIO 等のクラウドストレージに切り替えたい。なぜなら DB にファイルを保存すると容量と性能の問題が出るから。 | - |
| `Must` | 2 | 管理者として、`docker compose up` だけで全サービスを起動したい。なぜなら複雑な手順なしにすぐ動く環境を手に入れたいから。 | - |
| `Must` | 3 | 管理者として、すべての設定を `.env` ファイルだけで管理したい。なぜなら設定ファイルを何箇所も編集するのは間違いのもとだから。 | - |
| `Must` | 4 | 管理者として、`/api/health` にアクセスしてサービスの死活を確認したい。なぜなら監視ツールや Kubernetes からヘルスチェックできる口が必要だから。 | - |
| `Should` | 5 | 管理者として、DB とファイルストレージのバックアップ・リストア手順とスクリプトを使いたい。なぜなら障害時に素早くデータを復元できるようにしておきたいから。 | - |
| `Should` | 6 | 管理者として、バージョンアップ時のマイグレーション手順ドキュメントを参照したい。なぜなら更新作業を安全に進めてデータを失いたくないから。 | - |

## 受け入れ基準（Epic 完了の定義）

- [ ] 配下のすべての `Must` Story が Done になっている
- [ ] Docker Compose 起動・ストレージ切り替え・ヘルスチェックの E2E シナリオが通過している
- [ ] ドキュメント（ADR / API仕様）が更新されている
